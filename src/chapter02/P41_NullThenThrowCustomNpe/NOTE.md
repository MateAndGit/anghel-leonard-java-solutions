# [Java] Objects.requireNonNull: 더 우아한 Null 방어 전략

이 예제는 전통적인 `if (obj == null)` 방식에서 벗어나, Java 7부터 도입된 표준 유틸리티를 사용하여 코드의 가독성과 안전성을 동시에 높이는 방법을 보여줍니다.

---

## 1. OldCar vs NewCar 비교 분석

### ① OldCar: 고전적인 수동 검사
- **방식**: `if` 문으로 일일이 `null`을 체크하고, 수동으로 `throw new NullPointerException`을 호출합니다.
- **단점**: 코드가 길어지고 비즈니스 로직(데이터 할당)보다 유효성 검사 코드가 더 눈에 띄어 가독성을 해칩니다.

### ② NewCar: 현대적인 선언적 검사 (추천)
- **방식**: `Objects.requireNonNull(obj, message)`를 사용합니다.
- **장점**:
    - **가독성**: 한 줄로 검사와 할당을 동시에 처리할 수 있습니다.
    - **명확성**: "이 값은 반드시 필요하다(Require)"는 의도를 코드로 직접 드러냅니다.
    - **유지보수**: 메시지 처리가 정형화되어 실수를 줄여줍니다.



---

## 2. 왜 생성자와 메서드 입구에서 검사하나요? (Fail-Fast)

코드의 앞부분에서 `null`을 미리 차단하는 것을 **Fail-Fast(조기 실패)** 원칙이라고 합니다.

- **이유**: `null`이 시스템 깊숙이 침투하여 나중에 엉뚱한 곳에서 `NullPointerException`이 터지면 원인을 찾기가 매우 힘듭니다.
- **효과**: 문제가 발생한 즉시(생성 시점) 에러를 발생시켜 디버깅 시간을 획기적으로 줄여줍니다.



---

## 3. 실무 활용 사례

### [사례 1: 도메인 객체 보호]
사용자, 주문, 상품과 같은 핵심 도메인 객체를 생성할 때 필수 값이 누락되지 않도록 강제합니다.
- `this.userId = Objects.requireNonNull(userId, "사용자 ID는 필수입니다.");`

### [사례 2: 외부 라이브러리 연동]
외부 API나 라이브러리에서 전달받은 파라미터가 유효한지 메서드 시작점에서 즉시 검증할 때 사용합니다.

### [사례 3: 의존성 주입(DI)]
생성자를 통해 필요한 서비스 객체를 주입받을 때, 주입된 객체가 `null`이 아님을 보장하기 위해 사용합니다.

---

## 4. 실무형 요약 보고서

| 항목 | 고전적 방식 (`if`) | 현대적 방식 (`Objects`) |
| :--- | :--- | :--- |
| **코드 길이** | 3~4줄 소요 | **단 1줄** |
| **가독성** | 낮음 (검사 로직이 강조됨) | **높음 (의도가 명확함)** |
| **사용 버전** | 모든 Java 버전 | **Java 7 이상 권장** |

---

## 5. 현업 개발자의 유지보수 팁 📝

### ⚠️ 메시지의 중요성
`Objects.requireNonNull(name)` 처럼 메시지를 생략할 수도 있지만, 실무에서는 **"어떤 값이 null인지"** 명확한 메시지를 적어주는 것이 좋습니다. 에러 로그만 보고도 즉시 대응할 수 있기 때문입니다.

### 💡 Java 9의 선물: `requireNonNullElse`
자바 9부터는 `Objects.requireNonNullElse(obj, defaultValue)`가 추가되었습니다. `null`이면 에러를 던지는 대신 **기본값**을 쓰고 싶을 때 아주 유용합니다.



---

### 🧱 학습 메모
2장의 핵심 키워드는 **"견고함(Robustness)"**입니다.
- 입구에서 막는다 (**Fail-Fast**).
- 표준 도구를 쓴다 (**Objects.requireNonNull**).
