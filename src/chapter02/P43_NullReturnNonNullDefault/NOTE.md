# [Java] P43: Null 대신 안전한 기본값 반환하기 (Default Value Strategy)

모든 `null`이 에러는 아닙니다. 때로는 "이름이 없으면 'No name'으로 처리해"처럼 시스템이 유연하게 대처해야 할 때가 있습니다. Java 9부터 도입된 `Objects.requireNonNullElse` 시리즈가 이를 완벽하게 지원합니다.

---

## 1. 메서드별 상세 분석

### ① Objects.requireNonNullElse(obj, defaultObj)
- **원리**: 첫 번째 인자가 `null`이면 두 번째 인자인 기본값을 즉시 반환합니다.
- **특징**: "No name" 같은 단순한 문자열이나 이미 생성된 상수를 기본값으로 쓸 때 적합합니다.

### ② Objects.requireNonNullElseGet(obj, Supplier)
- **원리**: `null`일 경우에만 `Supplier`를 실행하여 기본값을 생성합니다.
- **특징**: `new Color(0, 0, 0)`처럼 객체를 새로 생성해야 할 때 사용합니다. **지연 로딩(Lazy)** 방식이므로, 데이터가 `null`이 아닐 때는 불필요한 객체 생성을 하지 않아 메모리를 아낄 수 있습니다.



---

## 2. '예외 던지기' vs '기본값 채우기' 선택 기준

실무에서 가장 고민되는 부분이죠. 언제 에러를 내고 언제 기본값을 줄까요?

| 선택 | 비즈니스 의미 | 예시 |
| :--- | :--- | :--- |
| **예외 던지기 (P41~42)** | "이 값 없이는 작업 진행이 **불가능**하다" | 회원 가입 시 이메일, 결제 시 금액 |
| **기본값 제공 (P43)** | "이 값은 없어도 되지만, **null은 위험하니** 안전한 값으로 채우자" | 프로필 사진(기본 이미지), 자동차 이름(No name) |

---

## 3. 실무 적용 시나리오

### [사례 1: UI/UX 레이아웃]
- 사용자가 닉네임을 설정하지 않았을 때 `null`을 화면에 띄우면 오류처럼 보입니다. 이때 `"익명"`이라는 기본값을 주어 시각적인 완성도를 높입니다.

### [사례 2: 설정(Configuration) 값]
- 시스템 환경 변수나 설정 파일에서 특정 값을 읽어올 때, 값이 누락되었다면 기본 타임아웃 시간이나 기본 경로를 할당하여 시스템이 멈추지 않게 합니다.

### [사례 3: 빈 컬렉션 반환]
- 리스트가 `null`인 경우 빈 리스트(`Collections.emptyList()`)를 반환하여, 호출하는 쪽에서 `for` 문을 돌려도 NPE가 나지 않게 배려합니다.

---

## 4. 현업 개발자의 유지보수 팁 📝

### ⚠️ 주의: 기본값도 null이면 안 됨!
`Objects.requireNonNullElse(obj, defaultObj)`에서 `defaultObj`마저 `null`이면 자바는 `NullPointerException`을 던집니다. 즉, **"마지막 보루는 반드시 진짜 값이어야 한다"**는 뜻입니다.

### 💡 Java 8 스타일 (Optional)
만약 Java 9 미만 버전을 쓰고 있다면 `Optional`을 사용하여 동일한 효과를 낼 수 있습니다.
- `Optional.ofNullable(name).orElse("No name")`

### 🧱 학습 메모
- **필수 데이터**는 입구에서 예외를 던져서 막는다 (**Strong Validation**).
- **선택적 데이터**는 기본값을 채워서 부드럽게 넘긴다 (**Soft Validation**).
