# [Java] P50: 가변 객체를 포함한 불변 클래스 설계 (방어적 복사)

P49에서 모든 필드가 기본 타입(`double`)인 경우를 다뤘다면, P50에서는 필드에 **가변 객체(Mutable Object)**가 포함되었을 때 불변성을 사수하는 핵심 기술인 **방어적 복사(Defensive Copy)**를 정리합니다.

---

## 1. 가변 객체 필드의 위험성

클래스를 `final`로 만들고 필드에 `final`을 붙여도, 필드가 참조하는 객체(`Radius`) 자체가 `setter`를 가진 가변형이라면 외부에서 내부 상태를 조작할 수 있는 구멍이 생깁니다.

* **문제점**: 생성자에서 받은 참조를 그대로 저장하거나, Getter에서 내부 참조를 그대로 반환하면 외부와 내부가 같은 메모리 주소를 공유하게 되어 불변성이 깨집니다.



---

## 2. 코드 분석: 방어적 복사(Defensive Copy) 적용

제공된 `Point` 클래스는 참조를 끊기 위해 입구(생성자)와 출구(Getter)에서 모두 복사본을 만듭니다.

### ① 생성자에서의 방어 (Incoming)
* 외부에서 전달된 `radius` 객체를 그대로 할당하지 않습니다.
* 새로운 `Radius` 인스턴스를 생성하고 내부 값을 일일이 복사하여 저장함으로써, 외부 원본과의 연결 고리를 완전히 끊습니다.

### ② Getter에서의 방어 (Outgoing)
* 내부에 저장된 `radius` 객체를 그대로 리턴하지 않습니다.
* 반환 시점에 다시 한번 복사본을 만들어 전달합니다. 이를 통해 호출자가 반환받은 값을 수정(`setStart` 등)하더라도 `Point` 내부의 실제 값은 보호됩니다.

---

## 3. 실무형 요약 보고서

| 단계 | 적용 기법 | 상세 동작 |
| :--- | :--- | :--- |
| **생성자 (Constructor)** | **방어적 복사** | 인자로 받은 가변 객체를 새 객체로 복사하여 필드에 저장 |
| **접근자 (Getter)** | **방어적 복사** | 필드 객체를 새 객체로 복사하여 외부로 반환 |
| **클래스 설계** | **참조 차단** | 외부와 내부가 동일한 가변 인스턴스를 공유하지 않도록 설계 |

---

## 4. 현업 개발자의 유지보수 팁 📝

### ⚠️ 성능 고려사항
* 매번 `new Radius()`를 호출하여 복사하는 것은 메모리 비용이 발생합니다. 하지만 데이터 무결성이 최우선인 보안/금융 도메인에서는 이 비용을 감수하고서라도 방어적 복사를 수행해야 합니다.

### 💡 대안: Java의 표준 도구 활용
* 필드가 컬렉션(`List`, `Map` 등)이라면 `Collections.unmodifiableList()` 등을 사용하여 수정 불가능한 뷰를 반환하는 방식을 고려할 수 있습니다.
* `Radius` 클래스에 복사 생성자(`public Radius(Radius other)`)를 구현해두면 `Point` 클래스의 복사 로직이 훨씬 간결해집니다.
