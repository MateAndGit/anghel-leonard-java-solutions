# [Java] P51: 빌더 패턴을 활용한 불변 클래스(Immutable Builder)

P51 예제는 복잡한 객체를 생성할 때의 편의성과 **불변성(Immutability)**을 동시에 챙기는 **빌더 패턴(Builder Pattern)**의 정석을 보여줍니다. 필드가 많아질수록 생성자만으로는 한계가 오는데, 이를 빌더가 어떻게 해결하는지 정리합니다.

---

## 1. 왜 빌더 패턴(Builder Pattern)인가?

필드가 5~6개 이상인 클래스에서 생성자만 사용하면 다음과 같은 문제가 발생합니다.
* **가독성 저하**: `new User("id", "pw", "name", "email", ...)` 처럼 인자가 많아지면 순서를 헷갈리기 쉽습니다.
* **선택적 인자 처리**: 어떤 유저는 이메일이 있고 어떤 유저는 이름이 없을 때, 모든 조합의 생성자를 만드는 것은 비효율적입니다.
* **불변성 유지**: Setter를 만들지 않으면서도 다양한 필드 조합을 유연하게 설정할 수 있습니다.



---

## 2. 코드 핵심 분석

### ① private 생성자와 정적 팩토리 메서드
* `User`의 생성자는 `private`으로 닫혀 있어 외부에서 직접 호출할 수 없습니다.
* 대신 `getBuilder(nickname, password)` 메서드를 통해 필수 값(Mandatory fields)을 먼저 강제하며 빌더를 시작하게 설계되었습니다.

### ② 내부 정적 클래스 UserBuilder
* `UserBuilder` 내부의 메서드들은 `return this;`를 반환하여 **메서드 체이닝(Method Chaining)**을 가능하게 합니다.
* `build()` 메서드가 호출되는 마지막 순간에만 실제 `User` 객체가 생성되므로, 생성 과정 중에는 값이 변할 수 있어도 완성된 `User`는 불변 상태가 됩니다.

### ③ 가변 필드에 대한 방어적 복사 (P50의 복습)
* `User` 클래스의 `created` 필드는 가변 객체인 `Date` 타입입니다.
* `getCreated()` 메서드에서 `new Date(created.getTime())`을 통해 **방어적 복사**본을 반환함으로써 외부에서 날짜 값을 조작하는 것을 원천 차단했습니다.

## 3. 실무형 요약 보고서

| 항목 | 구현 전략 | 장점 |
| :--- | :--- | :--- |
| **필수 필드** | 빌더의 생성자 파라미터로 배치 | 객체 생성 시 누락되는 데이터 방지 |
| **선택 필드** | 빌더의 체이닝 메서드로 제공 | 가독성 향상 및 유연한 객체 생성 |
| **객체 생성 시점** | `build()` 메서드 호출 시 | 불완전한 상태의 객체 노출 방지 |
| **방어적 복사** | Getter에서 가변 필드 복사 반환 | 완벽한 불변성(Immutable) 보장 |

## 4. 현업 개발자의 유지보수 팁 📝

### 💡 Lombok의 @Builder
실무에서는 이 많은 빌더 코드를 직접 짜지 않고, 롬복(Lombok) 라이브러리의 `@Builder` 어노테이션을 사용하여 단 한 줄로 해결합니다. 하지만 내부 동작 원리는 지금 보시는 코드와 동일합니다.

### 💡 필드 유효성 검사 (P41~45 활용)
빌더의 `build()` 메서드 내부에서 `Objects.requireNonNull()`이나 `Objects.checkIndex()` 등을 호출하면, 객체가 완성되기 직전에 모든 비즈니스 규칙을 최종 검증할 수 있습니다.
